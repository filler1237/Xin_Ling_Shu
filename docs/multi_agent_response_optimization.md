# 多智能体心理场景输出优化设计方案

> 目标：让最终输出既“像人说话”（共情、可读、可信），又“可执行”（有优先级、可验证），并让协同智能体真正完成推理整合而不是拼接。

## 1. 背景与现状

当前架构：
- A：解释/风险边界（症状判断、边界说明）
- B：行动方案（步骤化建议）
- C（仲裁）：输出最终回答

当前问题集中在最终回答的可读性、定制感和协同方式上：
1. 输出像指南/报告语言，缺少共情与“被理解”体验。
2. 缺少“症状 → 建议”的解释桥梁，用户感到通用模板。
3. 行动步骤并列堆叠，缺少优先级与执行负担控制。
4. 风险提示模板化，未结合严重程度与触发条件，用户容易忽略。
5. 协同智能体没有“推理整合”，更像“拼接+格式化”。

## 2. 设计目标（可验收）

以单次回答为单位，要求最终输出满足：
- **共情开场**：首段必须体现对用户状态的理解（不以“结合候选A/B”开头）。
- **解释桥梁**：至少给出 2 条“为什么这条建议适合你当前情况”的因果解释（短句即可）。
- **优先级清晰**：行动步骤必须有“先做哪两件事（最低行动集）”，其他归为“可选增强”。
- **风险提示个性化**：升级处理触发条件必须包含“持续时间 + 功能影响 + 伴随症状”至少两类。
- **协同可见**：C 的输出应融合 A 的风险边界与 B 的步骤，并对冲突/不确定性给出处理策略。

## 3. 输出结构（固定模板）

最终输出使用如下结构（强制）：

1) **先理解你（共情 + 复述）**
- 用 1–2 句话复述用户主诉/困扰（不下诊断）。

2) **我先判断一下风险边界（简短）**
- 风险等级（低/中/高）+ 一句理由 + 一句就医边界。

3) **为什么我建议你这样做（桥梁）**
- 建议 A 的原因（对应症状/目标）
- 建议 B 的原因（对应症状/目标）

4) **今晚/这周先做这 2 件事（最低行动集）**
- 每条包含：步骤、时长/频次、触发条件、验收指标

5) **可选增强（不超过 2 条）**
- 只在用户资源/精力允许时给出

6) **什么时候需要升级处理**
- 至少 3 条触发条件，且与当前严重程度相关

7) **依据来源（可选展示）**
- 列出关键依据来自哪些库/文件（2–4 条，避免刷屏）

## 4. 协同智能体（C）的“推理整合”设计

### 4.1 输入组织（给 C 的信息要可推理）

给 C 的输入不再只是 A/B 原文拼贴，而是结构化：
- 用户问题
- A 的三元组：`结论/关键依据/边界`
- B 的三元组：`行动目标/步骤/失败升级`
- 检索证据摘要：每条建议对应的来源片段（短句）
- 当前分析结果：意图、情绪、风险等级、危机标记

### 4.2 仲裁规则（不是“选A或B”，而是“融合并减负”）

优先级规则：
1. **安全优先**：若 A/B 任何一方提示高风险或不确定性大，最终回答必须保守。
2. **执行负担最小化**：默认只输出“最低行动集 2 条”，其余归为可选增强。
3. **桥梁强制**：每条行动建议必须附带一句“为什么适合你现在的情况”。
4. **冲突处理**：若 A/B 冲突，C 必须说明冲突点并给出“更安全的一侧 + 验证方式/观察指标”。

### 4.3 语言风格（像人说话）

风格约束：
- 禁止出现“结合候选A/B”“证据充分性评分”等报告语言。
- 用第二人称、短句、少术语。
- 先共情后建议：用户被理解后再给方案。
- 免责声明不使用模板段落，改成“就医边界 + 触发条件”。

## 5. A/B 的输出改造（让 C 更容易融合）

### 5.1 A（解释/风险边界）改造

强制输出：
- 只输出 3 段：结论、关键依据、风险边界
- 风险边界必须包含“观察指标（可量化或可描述）”

### 5.2 B（行动方案）改造

强制输出：
- 先给“最低行动集 2 条”（每条含：时长/频次/触发/验收）
- 再给“可选增强 ≤ 2 条”
- 最后给“失败升级路径（如果 3–7 天无改善怎么办）”

## 6. 个性化的“风险提示”设计

从分析结果与用户描述中抽取三个维度生成触发条件：
- **时间**：例如“连续 X 天/两周”
- **功能影响**：例如“影响上学/工作/进食/社交”
- **伴随症状**：例如“持续呕吐、体重下降、胸闷心悸、惊恐发作、自伤意念”

最终输出把这三维组合成 3–5 条短句，让用户更愿意读完。

## 7. 依据来源展示（让多智能体的“证据价值”可见）

展示策略（避免刷屏）：
- 只展示“最低行动集”对应的 1–2 条来源
- 以“来自：method_db / strategy_db 的《文件名/条目名》”形式呈现
- 不展示长片段，用户需要时再去“证据与运行状态”查看全文

## 8. 验收用例（最明显）

用同一个问题对比优化前后是否达标：
- 输入：用户描述“焦虑 + 失眠”，要求“今晚能做的方案 + 风险边界”
- 验收点：
  - 首段是否先共情复述
  - 是否有“为什么建议呼吸/20分钟规则”的桥梁
  - 是否明确“先做两件事”
  - 升级处理是否包含时间/功能影响/伴随症状至少两类
  - 是否没有报告语言（候选A/B、评分）

## 9. 实施清单（落地步骤）

1) 为 C 增加输出格式校验：不符合模板则回退到“结构化融合模板”（保证稳定）。
2) 改造 C 的提示词：强制共情开场 + 桥梁 + 最低行动集 + 个性化升级条件。
3) 改造 A/B 的提示词：输出结构固定，减少 C 的融合难度。
4) 在 UI 中给“仲裁最终版”增加一个可选开关：显示/隐藏“依据来源”（默认隐藏）。
